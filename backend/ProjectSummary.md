# Manufacturing Production Monitoring System - Backend Summary

## Project Overview

This backend is an ASP.NET Core Web API for a manufacturing production monitoring system. It manages users, machines, machine readings, maintenance logs, defects, products, production plans, production orders, production records, and production analytics.

The project follows a layered structure:

- `API`: controllers, authentication, authorization, middleware, background worker, admin audit-log endpoint
- `BusinessLayer`: service interfaces, service implementations, validators, business exceptions
- `DataAccessLayer`: EF Core database context and generic repository
- `Models`: entity models, enums, and DTOs
- `Tests`: xUnit unit tests for service behavior

## Improvements Added

- Added DTO request models for API create/update bodies.
- Removed server-controlled fields from request bodies where possible.
- Current user is now read from JWT claims for fields like `CreatedBy`, `ReportedBy`, and `EngineerId`.
- Current time is now set in services for fields like `CreatedAt`, `Timestamp`, `MaintenanceDate`, and `ProductionDate`.
- Controllers now use DTOs instead of exposing full entity models for POST and PUT requests.
- Validators are active again through service-layer validation.
- Duplicate POST checks were added before database insert. A duplicate means all meaningful non-id and non-timestamp fields match an existing row.
- Added a small in-memory audit log service.
- Added an admin-only endpoint: `GET /api/admin/logs`.
- Added fresh unit tests for Machine, Defect, and Product services.
- Fixed machine reading update mapping so route id is assigned to `ReadingId`, not `MachineId`.
- Fixed the machine GET role typo that had `"1Admin"` in the authorization string.

## Important API Design Notes

- Request DTOs are stored in `Models/DTOs/ApiRequestDtos.cs`.
- Entity models are still returned from GET endpoints.
- POST and PUT endpoints now accept DTOs.
- JWT tokens now include `ClaimTypes.NameIdentifier`, so controllers can identify the current logged-in user.
- Audit logs are only readable by users with the `Admin` role.

## Validation Rules

Validators check required fields, enum values, valid quantities, valid dates, price ranges, and text length.

Validation happens in the business layer, not inside controllers. This keeps controllers focused on HTTP work and services focused on business rules.

## Duplicate Prevention

Each create service checks existing data before inserting.

The duplicate check ignores:

- primary key fields
- timestamp/date fields that are generated by the server
- navigation properties

At least one meaningful field must be different from existing records before a POST can create a new row.

## Logging

Create, update, and delete actions write an audit entry with:

- UTC time
- logged-in username
- action name
- entity name
- entity id

The audit log endpoint is protected with `[Authorize(Roles = "Admin")]`.

## OOP Concepts Used

- Encapsulation: service classes hide validation, mapping, and repository calls.
- Abstraction: controllers depend on interfaces like `IMachineServices`, not concrete classes.
- Polymorphism: generic repository `IRepository<K,T>` works with multiple entity types.
- Composition: controllers receive services through constructor dependency injection.

## SOLID Principles Used

- Single Responsibility Principle: controllers handle HTTP, services handle business logic, validators handle validation.
- Open/Closed Principle: new DTOs and services can be added without changing the generic repository pattern.
- Liskov Substitution Principle: service interfaces can be replaced by mock or fake implementations in tests.
- Interface Segregation Principle: each entity has its own service interface.
- Dependency Inversion Principle: high-level controllers depend on abstractions through DI.

## DRY Techniques Used

- Reused generic repository for common CRUD behavior.
- Reused `DuplicateGuard` for duplicate-entry checks.
- Reused validators for both create and update workflows.
- Kept DTO mapping inside services to avoid repeating entity setup in controllers.
